Также опишите, какой логикой вы руководствовались в ходе принятия решений и выбора технологий. Не забывайте, что необходимо учесть все функциональные и нефункциональные требования.
Основываясь на нефункциональных требованиях, предложены:
1. Микрофронты и микросервисная архитектура бэкенда. API Gateway для оркестрации запросов и интеграции микросервисов. Apach Kafka для управления очередями запросов/сообщений
2. WAF и Nginx для защиты приложения от DDOS-атак, иньекций и других кибер-угроз
3. Интеграция с АБС минимальна, только в части обработки запроса открытия депозита (интеграция посредством Kafka)
4. Отдельно предложен микросервис СМС-нотификации в депозитном процессе
5. Предложены микросервисы для автоматизации работы менеджеров по ведению общих и индивидуальных ставок % по депозитам, продуктового каталога (депозиты и в будущем другие финансовые продукты)
6. Предложено решение для обеспечения доступности 99,9%: горизонтальное масштабирование, дополнительные зоны доступности, независимые кластеры Kubernetis, фейловер-стратегия Active-Active, репликация БД
7. Микросервисы на Java-стеке, микрофронты на React.js, СУБД PostreSQL (т.к. ипортозамещение)
8. Для защиты данных - передача данных по HTTPS (TLS 1.3), шифрование на уровне БД EAS-256
9. Мониторинг, логирование и аудит: Prometheus, Grafana, ELK, Jaeger. Сбор данных OpenTelemetry Collector
10. Для аутентификации клиентов предложен микросервис на keycloak. Для авторизации менеджеров - MS Active directory.
